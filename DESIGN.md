# DESIGN.md — 実装設計(seminar)

仕様の正は `docs/` と `db/schema.sql`。本書は「どの順で・どう作るか」の設計であり、
仕様の追加・変更はしない。仕様に無い判断は下部「確認事項」に集めた(CLAUDE.md の指示どおり
推測で進めない)。

## この作り方の価値(要約)

seminar は、より大きな構想——**業務・文書・メール(Microsoft 365 が担う領域)+
業務システム(基幹まで)+ 公開 Web(CMS が担う領域)を、OSS で AI とともに
自営する**——の一部(研修・セミナー業務)の実証である。詳細な考え方は
「自動化とAIの考え方」に、以下はその要約。

- **なぜ今できるか**: OSS はもともと無料で、塞いでいたのは組み立てと維持の
  労力・専門性だった。**AI がその労力を担う**ことで、自営が現実的になった
  (newly feasible)。「OSS を AI とともに作る」の"AI とともに"が決め手
- **三つを一つの土台で**: 共通土台(認証・SQL・OnlyOffice・メール・git・静的公開)で
  畳むと、ベンダー・契約・コストのサイロが消え、二重投資・継ぎ目・再入力がなくなり、
  **業務が一本の流れになる**(告知→申込→処理→記録→文書→連絡→会計)
- **標準でつなぐ**: 部品は独自形式や独自 API でなく**少数の公開標準**で疎結合に
  繋ぐ——データ=**Markdown / YAML / JSON / Parquet**、問い合わせ・蓄積=**SQL**
  (実装は PostgreSQL・SQLite・DuckDB。製品でなく SQL に commit)、処理・接着=
  **Python**。自己記述・AI が扱える・多ツール対応ゆえ、**部品は差し替え可能・
  データはツールより長生き・ロックインしない**(境界には CSV のような位置依存を
  使わず、自己記述/スキーマ付きを使う)
- **閉じた世界をつくらない**: 目的は、M365・業務パッケージ・CMS という閉じた
  世界(独自形式・独自 API・席数課金・データを外に出しにくい)を、別の閉じた
  世界(自前スタックへの新たなロックイン)に置き換えることではない。公開標準の
  上に築くことで、データは自由に出入りし、部品は差し替えられ、**この仕組み自体
  からも出られる**——開いた世界にする
- **所有は手元に**: ソフト(OSS 無料)・データ・そして**知識(ドメインルール)**まで。
  費用は利用者数に比例しない(クラウド AI = Claude は運用責任者 1 席で足りる)
- **責任構造とも整合**: 業務適合・法令遵守の最終責任は発注者にあり、透明・版管理・
  自己記述の器はその検証・監査・引き継ぎを自組織で回せる(§「自動化とAIの考え方」)
- **誰でもができる**: AI が authoring の敷居を下げ、専門家でなくても現場が
  安全な自動化・内容を持てる(属人化の解消)

正直な一線: これは **newly feasible(今こそ現実的)であって楽勝ではない**。
正しさの検証・段階移行・最終責任(発注者)は残る。AI は労力を桁違いに下げるが、
判断をゼロにはしない。

## 方針

- **様式 xlsx が唯一のフォーム定義**なので、実装の心臓部は
  「様式の生成(forms)と読み取り(パーサ)」。ここを最初に固め、
  ラウンドトリップテスト(生成した様式に記入→読み取り→元データと一致)で守る
- 3経路(ブラウザ記入/メール添付/簡易メール)は入口が違うだけで
  同一パーサ・同一登録処理に合流する。合流点(services)を先に作り、入口は後から足す
- 外部依存(IMAP/SMTP・OnlyOffice・PocketBase・Jitsi)はすべて抽象の裏に置き、
  開発時はフェイクで動かす。実環境の結線は導入フェーズの作業とする

## データフロー(合流の設計)

```
経路1 OnlyOffice callback ─┐
経路2 IMAP添付xlsx ────────┼→ parse.py(名前付きセル・版判定・検証)
経路3 簡易メール本文 ──────┘      │成功                │失敗
                            regist.py            IMAP「未処理」フォルダへ移動
                    (companies upsert・採番・    (原文ごと残る。黙って捨てない。
                     applications+attendees・     修正依頼の自動返信を送付)
                     received_file保存)
                            │
                        mail.py 受領メール(申込番号つき)

未処理キュー = IMAP フォルダそのもの(DB に写さない):
  INBOX(未着手)→ 処理済み / 未処理 / 差戻し の各フォルダへ移動。
  状態=所在フォルダ。事務局アプリは「未処理」を IMAP で一覧し、
  代行入力(source='staff')や差し戻しの操作でメールを移動する。
  経路1(callback)の読取失敗は編集画面へのエラー表示が主で、保存済み
  xlsx を添えて未処理フォルダへ投函し同じ列に合流させる。
  障害時は普通のメールソフトでフォルダを開けば全件見える(引き継ぎ容易)。
  FAX・紙はメールが無いので、キューを経由せず代行入力画面から直接登録
```

- parse.py は「xlsx → 申込データ(Pydantic)」の純関数に保つ(I/O なし)。
  経路3の本文読取も同じ申込データ型に正規化して regist.py へ渡す
- 検証(必須・参加場所の提供有無・期限・会場定員)は parse/regist の境界で行い、
  エラー種別ごとに自動返信文面を分ける(02_api の 3.)

## 実装フェーズ

**Phase 0: 足場**
pyproject(ruff/pytest)・core(設定・DB)・models(schema.sql と1:1)・
no.py(採番)+テスト。
開発用 PostgreSQL は後日結線(下記テスト方針)。models は接続なしで書ける。

**Phase 1: 様式(心臓部)**
forms.py: 講座ごとの様式 xlsx 生成(名前付きセル・入力規則ドロップダウン・
シート保護・様式バージョン&講座IDセル・記入例シート・FAX印刷レイアウト)。
parse.py: 名前付きセル読取・版判定・検証。
テスト: 生成→記入(openpyxl でセルに値を入れる)→読取のラウンドトリップ、
不備パターン網羅(必須欠落・期限切れ・定員超過・提供外の参加場所・旧版様式)。

**Phase 2: 登録と静的サイト**
regist.py(upsert・採番・received_file)、sitegen.py+site/
(分類別一覧・講座ページ・アーカイブ・QR・様式配布)。
ローカルでビルド→成果物を目視確認。Pages への反映は cf-publish を利用(→確認事項3)。

**Phase 3: メール(経路2・3)**
mail.py(送信抽象。開発はメモリ/ファイルへのフェイク、本番SMTPリレー)、
mailin.py(IMAP ポーリング・(a)添付/(b)登録済み送信者の判別・自動返信・
フォルダ移動〔処理済み/未処理〕)。経路3の本文読取は「講座特定+受講者名+参加場所」の
緩い抽出とし、確信が持てない場合はキューへ。
テストは IMAP をフェイク(メッセージを直接注入)して結合まで。

**Phase 4: 事務局アプリと帳票**
office/(Flet): ダッシュボード/講座管理(保存で sitegen)/申込一覧/
未処理受信(IMAP「未処理」フォルダの一覧・代行入力 source='staff'・
差し戻し=フォルダ移動)/企業一覧/一斉送信/名簿・実績/
エクスポート/スタッフ(admin)。roster.py・ledger.py(openpyxl 直接生成)。
PocketBase ログインは認証部を抽象化し、開発はスタブ(→確認事項4)。

**Phase 5: 経路1と導入一式**
routers/docs(セッション発行・callback、JWT)+site/ の編集ページ、
deploy/(systemd・Caddy・バックアップ手順・操作手順書の骨子)。
OnlyOffice Document Server 実機での検証は導入フェーズ。

各フェーズ完了ごとにコミット(機能単位・日本語メッセージ)。

## テスト方針

- pytest。parse/forms/no/ledger/roster は純粋なロジックとして厚めに
- DB を使う結合テストは PostgreSQL 前提(TIMESTAMPTZ・gen_random_uuid)。
  **開発用 PostgreSQL の用意は後日**のため、DB 結合テストはマーカー

## 決定事項(2026-07-07 確認済み)

1. **未処理キュー = IMAP フォルダ**(DB テーブルは追加しない)。
   メールボックス自体をキューとして使い、状態は所在フォルダで表す
   (INBOX→処理済み/未処理/差戻し)。schema.sql は変更なし。
   原文はメールサーバーに原文のまま残り、メールソフトでも確認できる。
   処理の操作者記録は applications 側(source='staff'+事務局メモ)で足りる
2. **開発用 PostgreSQL**: 後日。それまで DB 結合テストはスキップ運用(上記)
3. **申込番号**: 採番方式を設定(SEMINAR_NO_STYLE)で切替可能にする
   (2026-07-08 更新。サンプル・教材としての価値も兼ねる——規模に道具を合わせる実例)。
   `seq`=全体通し連番(既定。例 2026-00042)/ `fy`=年度リセット /
   `fy-cat`=年度-分類-何回目(例 2026-DX-3。番号だけで内容がわかる小規模向け)。
   実装は no.py に閉じる
4. **PocketBase・OnlyOffice**: 開発中はスタブ、Phase 5 でまとめて実物結線
5. **様式・宛先の非公開**(2026-07-08): 公開サイトから form.xlsx と
   申込アドレスを撤去。様式と宛先は受領メール・印刷物・電話で個別に渡す
   (ボット収集・バックスキャッター対策)。添付なし・未登録送信者への
   自動返信(no_form)は廃止し、未処理フォルダ行きのみに。
   公開サイトの連絡先は電話等の公開用文言(SEMINAR_CONTACT_NOTE)
6. **様式の発行キー**(2026-07-08): 申込者にパスワードを持たせない代わりに、
   様式へ HMAC(SEMINAR_FORM_SECRET, 講座ID+様式版)の発行キーを
   非表示セルで埋め込み、パーサで検証(保存不要・ステートレス)。
   検証不可=発行物でない様式は Invalid → 未処理キューへ。
   secret 未設定時は検証オフ(開発用)。運用では必須の環境変数とする
7. **送信用テキスト方式**(2026-07-08、ユーザー発案): 記入内容+発行キー
   から `項目: 値` 形式の本文を生成→メール本文に貼って送信(添付不要・
   ゆれなし)。**推奨は OnlyOffice マクロ**(未記入チェック付き・無料の
   Desktop で可。チェック定義はサーバー検証と同じ forms.py から自動生成
   ——様式が複雑になっても数式と違い破綻しない)。**数式は当面の
   フォールバック**(1行1項目・全表計算ソフト共通・チェックなし)。
   正の検証は常にサーバー側(parse)。パーサは parse.py に集約(parse=xlsx / parse_text=本文、
   検証は共通 _assemble)。マクロ JS は forms.py から自動生成
   (`python -m app.services.forms`)。**マクロの xlsx への自動埋め込みは
   実機(OnlyOffice)検証後**——それまでは導入時に手動で組み込む
   (deploy/form-macro.js)。実行時に LLM は使わない
   (生成は機械、ゆれは発生源から断つ)

## 認証・本人性の考え方(2026-07-08 整理)

「アカウントを持たせない・個人情報を Web に出さない」(CLAUDE.md / 決定5・6)の
背景。**チャネルはそれぞれ別のものしか証明しない**ので、「何を証明する必要が
あるか」で手段を選ぶ。証明できる以上のものを課す手段は、コストと負債になる。

| 手段 | 証明するもの | 本システムでの扱い |
|---|---|---|
| パスワード | 秘密の知識 | **使わない**。一回性の申込者に保管パスワードを持たせると、最大の負債(侵害の的)を最小の便益で抱える |
| メール引き当て | メール箱の支配 | 申込の引き当てはこれのみ(決定5)。流れる価値が低い(申込1件)ため被害範囲が小さく許容 |
| 発行キー(HMAC) | その様式が発行物であること | 様式にステートレス埋め込み(決定6)。URL でもパスワードでもなく**現物(xlsx)に埋めた capability** |
| 郵送 | その住所で受け取れること | 下記。住所・居所の紐付けが要る講座だけ |
| 対面受付 | 当日その場に居ること | 会場での本人確認が要る場合のみ(最強・最も高価) |

**メールは厄介**: 送信者も添付も内容も定義できず、性善説の SMTP を後付けで守る
チャネル。だから重要なものを運ばせない——申込の引き当て(低価値)には使うが、
個人情報の表示・プリフィルには使わない(CLAUDE.md)。様式・宛先も公開しない(決定5)。

**郵送=住所確認の道具(必要な講座だけ)**: メール引き当ては「メール箱の支配」までしか
言えず、住所・居所は証明しない。自治体の住民限定講座・補助要件・修了証や受講票の
現物送付など、**住所の紐付けが実際に要る場面でのみ**郵送を差す。専用の確認コード
往復は作らず、**送る現物(受講票・修了証)を申告住所へ送ることが検証を兼ねる**
(届いた=住所は実在 / 宛先不明で戻る=悪い住所を捕捉)。
留保: 証明できるのは「その住所で受け取れる誰か」であって本人そのものではなく、
数日の遅延がある。**投機的に作らない**——住所紐付けを要する講座が実在してから
per-course で足す(仕様変更は安い)。

**守る本丸は玄関でなく名簿**: 認証方式は入口の話。漏れて困るのは受講者名簿・
実績台帳(個人情報)。力はそこに注ぐ——office 側のアクセス制御(DB直結・LAN)、
台帳の保存時の扱い、そして**保持期間と削除の方針**(終了した講座の参加者情報を
いつ消すか)。**保持期間・削除は運営者(機関)の責任**(2026-07-08 決定)——
機関の記録管理・個人情報保護の規程に従う。システムは削除の手段を用意するが、
期間を決め打ちしない(規程は機関ごと・講座ごとに異なるため)。

## 自動化とAIの考え方(2026-07-08 整理)

申込様式の自動化(送信用テキストの生成・読み取り)をどう作るか、その道具に
AI をどう使うかの判断枠組み。docs/05(AI活用)・docs/07(なぜ M365 か)と地続き。

**VBA を使わない=様式は仮でなく本格**: 様式の自動化は VBA ではなく、役割を
分離して作る。**外部 Python(処理)**=様式・本文の読み取り・検証・登録
(services/parse.py。文書の外・git 管理・監査可能。開いても起動しない)、
**内部 JavaScript(補助)**=OnlyOffice のマクロ(サンドボックス。送信用テキストの
組み立てだけ。無くても数式・添付のフォールバックで成立)。データと実行コードが
同居しないので、「文書を開く=特権実行」という VBA の攻撃経路がない。神Excel は
その場しのぎではなく、この分離ゆえ最も硬い部品。

**誰でもができる(VBA も属人化していた)**: 手で書くなら VBA も Python・JS も
簡単ではない——VBA も書ける一部の人に属人化し(「開かずのマクロ」)、危険でも
あった。AI が手書きのハードルを取り払い、専門家でなくても現場の担当が自分で
安全な自動化を作れるようになる。難易度の比較ではなく、**属人化の解消と裾野の
拡大**が本質。

**AI の使い分け(用途 × 機微)**: docs/05 の二層を具体化する。
- コード: 基本は Claude(パーサ・生成器・様式マクロの基盤 = forms.py の
  macro_js 等。非機微)。細かい調整・現場の小さな自動化・内部データに触る分は
  North Mini Code(ローカル・VBA の代替)
- 文書: 基本は Claude(公開情報の変換・下書き)。内部秘・個人情報を含む分は
  Command A+(ローカル)
- 原則: 公開・非機微 → クラウド(Claude)/ 内部秘・個人情報・現場の局所 →
  ローカルで建物の外に出さない
- AI は開発・運用の道具。出力(Python・JS・Markdown)は人が確認して git に
  凍結し、公開物・処理経路にはランタイムの AI を残さない(parse は AI ではなく
  凍結された通常のコード)

**コスト**: 部品はほぼ OSS/オープンウェイト(継続ライセンス料なし)。クラウド
(Claude)が要るのは構築・移行を回す運用責任者だけで、職員全員に配らない
(1席で足りる=M365 の席数課金の逆で費用が人数に比例しない)。OSS ゆえ調達・
契約のリードタイムがなく基盤の立ち上げは速い。構築・移行の手間の大半も AI が
担う(人に残るのは確認・判断・最終責任)。docs/07 の M365 批判の裏返し。

**表計算を薄くする(データ=宣言的 / 計算=Python / 体裁=テンプレート)**:
Excel の多重関数リンクは、データ・ロジック・体裁をセルの中で絡ませる——壊れ
やすく・不透明・版管理もテストもできない。重い集計はピボットやアドインに
閉じ込められ、席数ライセンスと属人化を生む。**この分離を邪魔しているのが
Excel(表計算を"アプリ本体"にする使い方)**。移行後は表計算を薄くする:

- データ → **YAML/JSON**(宣言的・git で差分が見える・行移動で壊れない)
- 計算 → **Python**(監査・テスト可能・限界費用ゼロ。ピボット/アドインの代替。
  seminar は roster.py・ledger.py の openpyxl 集計で既に体現)
- 体裁 → **テンプレート**(生成物として分離)
- 表計算(OnlyOffice)は「**記入・表示**」に軽くする。ロジックの置き場にしない

橋渡しはマクロ: OnlyOffice の JS マクロが記入済みシートを構造化 JSON/YAML に
変換する。**マクロは AI(North Mini Code)が開発時に生成し、runtime は決定的な
素の JS**(ランタイムに AI を置かない)。forms.py の macro_js が様式定義から
マクロを自動生成しているのがこの形。流れ: 記入(OnlyOffice)→ マクロ(AI製・
決定的)→ JSON/YAML → Python・テンプレート。

**システム間の受け渡しは自己記述形式(CSV を避ける)**: 収納・経理など別システムへ
データを渡す/入力する場面(有料講座の受講料・手数料の収納など。決済そのものは
アプリに持たず外部に委ねるが、渡すデータのインターフェースは要る)。ここで
**CSV は項目が変わると位置がずれる**(黙って壊れる)ので使わない。名前と型が
データに付く形式にする——小さければ JSON/YAML、**人数が多い/大量なら Parquet**
(列指向・型付き・**項目名で判断できる**・DuckDB 等でそのまま読む)。
**経理関係のシステムも Parquet で入力できるようにしておく**と、受け渡しの両端が
自己記述になり、項目の追加・変更で壊れない。有料講座・多人数は現実にあるので、
この受け渡しインターフェースは(規模が要求すれば)設計に持つ。

**正しさ(業務適合・法令遵守)の最終責任は発注者にある**:
システム開発の委託は請負(民法632条)または準委任(同656条)であり、受託者
(ベンダー)が負うのは契約で定めた仕様への適合——請負では仕事の完成と契約
不適合責任(同562条以下)、準委任では善管注意義務(同644条)——にとどまる。
要件・業務要件の確定は発注者の責任領域であり(発注者には判例上の協力義務が
ある)、仕様が自組織の業務に適合しているかは発注者が定める。さらに、帳簿・
給与等の法令遵守義務(電子帳簿保存法・法人税法・労働基準法等)は、それらを
作成・保存する事業者=発注者自身に課される義務であって、ベンダーは遵守義務の
名宛人ではない(専門家としての助言・説明義務にとどまる)。したがって業務
ルールの妥当性と法令適合の最終責任は発注者にあり、これをベンダーの保証に
委ねることは責任構造に反する。

ゆえに、ルールを宣言的データ+コードで表現し git で版管理し、**発注者自身が
検証・保守・監査できる形にすることは、コスト以前に責任構造の要請**である
(法令は公開情報で、encoding は AI が支援できる)。透明・版管理・自己記述で
手元に保つことで、監査(版履歴が証跡)・法改正対応(差分で更新・レビュー)・
引き継ぎが自組織で回る。器(PostgreSQL+Python+宣言的データ)は小さな業務
システムから基幹まで同じで、規模が上がるほど厳密さ(検証・可用性・法適合)を
上に振る——器は同じ、賭け金が高い。自組織が自分の業務ルールを把握している
こと自体が、正しさとガバナンスの土台。
注記: 契約類型・責任範囲の具体は個別契約と適用法令の確認による(本項は
一般的な責任構造の整理であって法的助言ではない)。

**結果として業務システムは簡単になる**——複雑さを管理するのではなく、
名前付きデータで繋いだ単純な部品に分解して消すから。各部品は一役で境界が明確
(データ=宣言的/計算=Python/体裁=テンプレート/接続=自己記述)。だから
分かる・監査できる・直せる・引き継げる。難所は AI が書き、人が向き合う業務
ロジックは小さく宣言的なまま。「簡単」は部品数が少ないことではなく、各部品が
単純で結合が緩いこと——立っている状態が簡単、が正確(構築は一度きりで大半は AI)。
