# 04 インフラ・デプロイ(seminar)

**機関内のローカルサーバーに導入する(1機関1式)。VPS・外部クラウドは
使わない**——申込者・受講者の個人情報、会議、文書は機関の建物の中に置く。
外部に出すのは個人情報を含まない公開講座情報の静的配信のみ。

## 構成(自営スタック)

- アプリ機(機関内): FastAPI・PostgreSQL・PocketBase・事務局アプリ(Flet)・
  OnlyOffice Docs(Document Server。ブラウザ記入と事務局の文書編集)・
  Forgejo。Caddy がリバースプロキシ(/api→FastAPI、
  /auth→PocketBase、/flyers→file_server)。計画停止・メンテ可
- ローカルAI(機関内・段階導入): Ollama 等の実行環境に Command A+
  (文書業務・紙アンケート処理)と **North Mini Code(小規模 Python の
  生成。VBA の代替)**を載せる。機材要件・導入時期は docs/05。
  アプリ機と分けるか、メモリに余裕のあるマシンに同居
- **配信機(機関内): Jitsi Meet**(オンライン受講の自営配信。Zoom を
  置き換える)。配信中は帯域と CPU を使うため、アプリ機と分けるか、
  余裕のあるマシンに同居させる。同時接続数は回線の上り帯域に依存する
  ため、初回は小規模講座で検証してから本番に使う(移行期は Zoom 併用可。
  meeting_url は中立な URL として設計済み)。録画が必要な場合は
  Jibri を追加(要検証)
- 公開静的サイト(講座情報・ブラウザ記入ページ)は Cloudflare Pages 配信。
  OnlyOffice Docs(Document Server)はアプリ機のコンテナで稼働し、
  Caddy 経由で公開(JWT 必須)。編集セッション中のみアプリ機に依存する
  (停止中は電話・紙へ案内。既知企業はメールで受け付けられる)
  (アプリ機停止中も講座の閲覧は可能。申込メールは
  受信箱に溜まり、アプリ機の復帰後に mailin が順に処理する——
  メールの再送特性がそのまま停止耐性になる)
- Cloudflare: DNS+プロキシ(Full Strict・Origin CA)。443 は
  Cloudflare IP レンジのみ許可。**固定IPが必要**(公開・メール・配信のため。
  機関の既存回線の契約を確認)
- OnlyOffice: 事務局の文書・台帳編集(xlsx は本システムが直接生成し、
  OnlyOffice で開く)。Forgejo: 本システムのコード・設定・手順書の置き場
  (引き継ぎの受け皿。機関の IT 人材が中を見られる状態を保つ)

## メール(重要)

**申込の受信(mailin)**・受領返信・一斉送信が業務の生命線。申込専用
アドレスの受信箱を IMAP で監視する(既存メール環境にアドレスを1つ追加)。
**このアドレスはページ本文に書かない**(様式 xlsx の中・受領メール・
印刷物にのみ記載。ボット収集対策。様式由来でないメールに自動返信しない
ことでバックスキャッターも防ぐ)。
送信元は機関の既存ドメインである
必要があるため、**段階1は機関の既存メール環境(SMTP)をリレーとして利用**
(SPF/DKIM の整合を崩さない)。services/mail.py は送信手段を抽象化してあり、
**段階2で自営メールサーバー(Stalwart)へ移行**する(固定IP・PTR・
SPF/DKIM/DMARC の設定が前提。メール機は常時稼働とし、アプリ機と分ける)。
一斉送信は連続投函でなく間隔を空けて送る(先方サーバーの受信制限対策)。

## バックアップ・運用

- 毎晩: pg_dump+チラシPDF+PocketBase データを rclone でオフサイトへ。
  保持14日。個人情報を含むためバックアップ先も機関管理の領域とする
- systemd 常駐。デプロイは git pull+restart(短時間停止は許容。
  申込期限直前の時間帯を避ける)
- 監視: Cloudflare Health Check(5分間隔・メール通知)。メールも、止まると
  「来るものが来なくなるだけ」で内側からは気づけないため、自分宛て送受信の
  疎通確認を定期実行して失敗を通知する(小さなスクリプトで足りる。
  それ以外のサービスは止まれば使う人が気づくので専用監視は置かない)

## WordPress からの段階移行

1. 新システムを別サブドメイン(例 kensyu.〜)で稼働開始
2. 既存 WordPress のセミナー記事から新講座ページへリンク(並行稼働)
3. 新規講座は新システムのみで告知。旧記事は残す
4. 落ち着いたら、他カテゴリのページを静的化して順次移設
5. 全移設後に WordPress を退役(更新・プラグイン保守・脆弱性対応の
   負担が消える)。既存URLは Cloudflare のリダイレクトで維持

## 引き継ぎ(運用の移管)

構築後、機関の IT 人材(ITコーディネーター等)へ運用実務を研修し、
運用責任を移管する(研修は時給1万円×実時間)。

- 対象実務: 講座登録〜募集開始・静的サイト再生成、申込一覧と定員の見方、
  一斉送信、未処理受信キューの処理(読取失敗・FAX の代行入力)、
  当日名簿の xlsx 出力と出席者数入力、台帳 xlsx 出力、
  紙アンケートのローカルLLM処理(docs/05)、
  ローカルAIでの集計プログラム生成の演習(North Mini Code+OpenCode。
  「かつて VBA でやっていた自作を、この環境でやる」)、確認メール再送、
  バックアップの確認、リストア手順の演習
- 成果物: office/(事務局アプリ)の操作手順書、障害時の一次対応手順、
  デプロイ手順(git pull+restart)
- 到達目標: 日常運用と講座サイクル(告知→申込→開催→実績)を
  機関の人材だけで一巡できること。当方への依存を残さない
