# 07 なぜ Microsoft 365 を採用しないか

本システムが扱う処理(申込 xlsx の読み取り・名簿・台帳の生成・小規模な
自動化)を Microsoft 365 で実現しようとすると、取り得る道は二つしかない。
**どちらも構造的な欠陥を持つ。** これは価格の問題ではなく、作りの問題である。

## 道1: ローカルで自動化する = VBA(マクロ)

Office の中でローカルに自動化する手段は VBA しかない。

- マクロ付き Office 文書は、長年サイバー攻撃の主要な侵入経路であり
  続けてきた(Emotet をはじめ、マルウェアの標準的な運搬手段)
- その結果、Microsoft 自身が 2022 年以降、インターネット由来の文書の
  マクロを既定でブロックするに至った。**ベンダー自身が自社の自動化手段を
  危険物として封じた**、という事実がすべてを物語る
- 根本原因は、データ(文書)と実行コード(マクロ)が同じファイルに
  同居する設計にある。文書を開くことがコードの実行になり得る構造は、
  修正不能である

使えば脆弱、封じれば自動化できない。ローカルの道は公式に閉じられている。

## 道2: クラウドで処理する = Power Automate / Copilot / Graph API

ローカルが封じられた結果、Microsoft はクラウドでの処理へ誘導する。
しかしこの道は三つの点で不合理である。

1. **データの外部送出**: 申込企業の名称・担当者・住所・電話といった
   実データを、手元の集計のために外部データセンターへ送ることになる。
   なお「大手クラウドに預ければ安全」という前提は成り立たない——
   Microsoft のクラウド基盤自体への侵害は現実に発生している
   (2023年 Storm-0558 による Exchange Online 署名鍵の窃取、
   2024年 Midnight Blizzard による社内メール侵害)
2. **エネルギーの浪費**: 「セルを読んで集計する」という手元のPCで
   数秒・数十ワットで済む処理のために、データセンターまでの往復と
   その電力を動員する。処理の実体に対して動員される資源が桁違いに
   不釣り合いである(評価)
3. **利用者数に比例する月額**: 上記の仕組みの利用料として、
   Copilot を含む各機能に利用者ごとの継続課金が発生する。
   処理の中身は変わらないまま、支払いだけが恒久化する

## Copilot について

Copilot の価値提案は「Office の中に AI がいる」ことである。しかし AI が
Office の中に居る必要があったのは、**データが Office のクラウドの中に
囚われていたから**にすぎない。データがただの xlsx ファイルとして手元に
ある本構成では、AI を Office に同居させる理由そのものが消える。
文書業務の AI は Command A+(建物内)、小規模自動化の生成は
North Mini Code(建物内・VBA の代替)が担い、追加の月額は発生しない。

## 費用の具体像

- Microsoft 365 は利用者ごとの月額課金であり、値上げが繰り返されている
  (2022・2023・2024〔20%〕と改定が続き、2026年7月1日にも価格改定。
  例: E3 は 5,197円→5,597円/月・税抜。Business Premium は5年間で
  2,180円→3,298円と 51% 上昇)
- 業務に必要な水準のプランは実勢で**1人あたり月5,756円程度**
  〔プラン名を確認して記載〕。年 69,072円/人となり、
  職員30人で**年約207万円**、50人で**年約345万円**が恒久的に発生する
- 本構成: ライセンス費ゼロ(AGPL / Apache 2.0)。任意の運用支援は
  **組織全体で月1万円**(年12万円)。M365 の**1人分の年額の約2倍で、
  組織全体の支援が賄える**。構築・引き継ぎは一回性の費用(時給1万円×実時間)

## 自動化は別料金(従量課金の二段構造)

Microsoft 365 の席数定額が覆うのは**手作業の範囲**である。本システムが
扱うような集計・処理を「自動化」しようとした瞬間、課金の性質が変わる。

- Power Automate: 標準の簡易フローは 365 に含まれるが、実務に必要な
  水準(プレミアムコネクタ・無人実行〔RPA〕・高頻度実行)は別建ての
  プラン(per-user / per-flow の月額)になる。文書を AI に読ませる
  AI Builder はクレジット消費(従量)
- Python 等で処理したければ、Microsoft の世界では Azure Functions /
  Logic Apps へ誘導される。これは **Azure サブスクリプション=純粋な
  従量課金**(実行回数・実行時間・ストレージ)であり、365 とは別の契約・
  別の管理体系(Azure portal)になる
- Copilot のエージェント的な利用(Copilot Studio・Agent 365 等の
  長時間自動作業)にも、月額の上に処理量ベースの消費課金が重なる。
  これらを束ねた上位プラン E7 は、個別合計で月 117 ドル相当とされる水準
  である

つまり実像は「**手作業は定額、自動化は従量**」の二段構造である。
席数料金は入場券にすぎず、仕事を機械に任せるほどメーターが回る。
従量課金は席数と違い**予算を事前に確定できない**。組織の経理にとっては
金額の多寡以上にこの点が致命的であり、結局「自動化機能は使うな」という
運用に帰着しがちである(評価)。

そして従量で買っている処理の正体は、ただの Python である。本構成が
機関内のマシンで**限界費用ゼロ**(電気代のみ)で走らせる openpyxl の
集計と、Azure がミリ秒単位で課金しながら走らせる処理は、同じものである。
エージェント型 AI も同じ論理で、ローカル(North Mini Code / Command A+)
に置けば、一晩働かせても増えるのは電気代だけである。

## 本構成は二択の外側にある

| 観点 | M365(道1: VBA) | M365(道2: クラウド) | 本構成 |
|---|---|---|---|
| コードとデータ | 同居(文書を開く=実行) | — | 分離(xlsx は保護とデータのみ。処理は独立した Python) |
| データの所在 | 手元 | 外部データセンター | **建物の中** |
| AI | — | Copilot(クラウド・月額) | Command A+ / North Mini Code(ローカル・追加費用なし) |
| エネルギー | 手元 | データセンター往復 | 手元の数十ワット |
| 継続費用 | ライセンス月額 | ライセンス+AI 月額 | なし(自営) |
| 自動化の費用 | —(封鎖済み) | 従量課金(Power Automate / Azure / クレジット。青天井) | **限界費用ゼロ**(ローカルで Python が走るだけ) |

「ローカルは危険、クラウドは安全で高い」という前提は、VBA という
欠陥設計を前提にした場合にのみ成立する。データにコードを同居させず、
処理を独立した Python(openpyxl)に置き、AI をデータのある建物内に
置けば、**ローカルこそが安全で、安く、速い。**
